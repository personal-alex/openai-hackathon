# Codex Collaboration Evidence Instructions

Read this file when a task materially contributes to hackathon delivery or when
preparing submission evidence.

## Primary build thread

Use one primary Codex thread for the majority of core implementation work:
scaffolding, contracts, event packs, compiler, tests, product integration,
deployment, and submission preparation.

Submit `/feedback` from that primary thread near completion. Record the returned
session ID exactly in `docs/CODEX_COLLABORATION.md` and the submission checklist.
Do not substitute a visible thread ID for the `/feedback` result.

## When to log

After material Codex-assisted work is completed and verified, append a concise
entry to `docs/CODEX_COLLABORATION.md`. Material work includes accepted planning
or HLD/UX artifacts, scaffolding, implementation, test generation/repair,
meaningful debugging, deployment, or durable decisions.

Do not log routine reads, speculative discussion, trivial edits, unapproved
proposals, or failed experiments.

## Required entry fields

- Date
- Primary Codex thread/session reference when available
- Task or outcome
- Codex contribution
- Human review, decision, or modification
- Changed artifact, commit, pull request, or Linear issue
- Verification evidence: command/test result, demo evidence, or review outcome

Do not claim a Codex contribution that cannot be supported by thread history,
repository changes, commits, tests, or demo evidence. Never include secrets,
API keys, personal data, raw user stories, or sensitive content.
